01Executive summary

OB-1 is a terminal-first coding agent. Its default is a single, fast, deeply token-optimized agent (Solo) — because the research is blunt: at equal token budgets, single agents match or beat multi-agent systems on most coding work, and multi-agent systems cost 4–15× more (R5). The differentiator is that OB-1 can escalate into one of three collaborative modes when a task is hard or high-value:

Fusion

Best-of-N

Fan the same task to several models in isolated contexts, then synthesize one final answer from the strongest parts of each.

Council

Critique & converge

Models draft, then adversarially critique each other across rounds until a vetted answer survives.

Personas

Role panel

Dynamically formed expert personas (Architect, Security, Performance, Skeptic…) collaborate and a lead synthesizes.

All three sit on top of a shared, best-practice foundation: plan/act loop, diff-based editing, a tree-sitter repository map, tiered on-demand memory, MCP tools + skills, OS sandboxing, a multi-provider gateway, and a built-in token/cost meter.

03System architecture

Eight layers. The top three are what make OB-1 distinctive; the bottom five are the best-practice foundation every modern agent is converging on.

04The core agent loop

Every mode — even multi-mind workers — runs this same gated loop. Plan/Act separation (pioneered by Cline, R6) and an approval gate before any side effect are non-negotiable.

05Context & memory engine

This is where most of the token savings — and the "very visible" memory — live. Treat the context window like a cache you fill lazily: a tiny always-loaded index, everything else externalized and pulled in top-k on demand (R3 — Claude Code's MEMORY.md pattern). Long-term memory follows Google Memory Bank's model (R8): each item is a plain-text fact with immutable revisions (readable + auditable), indexed by sqlite-vec for retrieval — plus a relationship graph over the entities (next section).

Why this is the highest-leverage subsystem. R3's efficiency numbers (90%+ token savings from selective retrieval, 91% lower latency) are structurally robust — selective top-k mechanically beats reprocessing full history. We bank these near-tautological savings on day one, before touching any model.

Relationships — a lightweight, visible property graph

Memory isn't just isolated facts; OB-1 also stores entities and the typed relationships between them — LoginController —calls→ AuthService, AuthService —issues→ JWT — as a property graph in the same SQLite file as the fact store. Following Graphiti's bi-temporal model (R8), each edge carries a validity window and is invalidated, not deleted when things change, so OB-1 can answer "what did the auth flow look like three refactors ago?" The graph renders directly in the /memory viewer (or an HTML / Neo4j export) — this is the "very visible" part.

The graph cost-control rule (R8)

Graph storage is cheap, but graph-heavy retrieval is what cost Mem0's graph variant ~2× tokens and ~3× latency for only ~2% accuracy. So OB-1 stores the full graph and visualizes it, but by default injects only top-k facts + a bounded 1–2 hop neighborhood — never the whole graph. Full multi-hop traversal is opt-in, for queries that genuinely need temporal/relational reasoning.

06The four modes

One workhorse and three escalations. The CLI default is Solo; users switch with /mode fusion|council|personas or OB-1 can suggest an escalation when it detects a hard/high-value task. Each mode declares a token budget up front so cost is never a surprise.

Fusion  Best-of-N synthesis

The same fully-specified task is dispatched to several models (and/or the same model at different temperatures/strategies) running in isolated context windows. Each returns a complete candidate. An aggregator — a strong synthesizer model — reads all candidates plus their diffs/test results and produces one final answer, grafting the best parts of each. Optionally each candidate is auto-scored (compile/tests) so the synthesizer is grounded in real signal, not vibes.

When to use: hard, high-value, parallelizable tasks (a tricky algorithm, an ambiguous design). Honest caveat (R5): cost scales ~N×; much of the benefit of "more agents" correlates with raw compute, so Fusion must be compute-matched against Solo in the eval harness before it's recommended by default.

Council  Critique & converge

Models produce drafts, then each adversarially critiques the others — prompted to refute, not flatter — across one or more rounds. Authors revise against the critiques; an arbiter checks whether objections are resolved and declares convergence (or escalates to the user). To respect the research, critics are given distinct lenses (correctness, security, performance, does-it-actually-reproduce) rather than N identical reviewers — diversity catches failure modes redundancy can't.

When to use: correctness-critical or risky changes (auth, migrations, money). Honest caveat (R5): coordination/critique adds a distinct failure class and real cost; a critic only helps if it's genuinely better than the generator — so Council uses strong models as critics and caps rounds.

Personas  Dynamic role panel

OB-1 inspects the task and forms the personas it actually needs — e.g. Architect, Security Auditor, Performance Engineer, API Designer, and a standing Skeptic. Each works its angle and posts findings to a shared blackboard (durable shared state, which R5 names as the antidote to multi-agent fragility). A lead persona integrates the board into a final plan/answer. This is MetaGPT's role pipeline (R6) made dynamic instead of a fixed five-role waterfall.

When to use: open-ended design, architecture reviews, "what am I missing?" Honest caveat (R5/R6): rigid role pipelines can over-engineer simple tasks; the Former is allowed to pick one persona (collapsing to Solo) when the task doesn't warrant a panel.

07Token-optimization strategy

Two buckets: structural savings (always on, near-tautological) and behavioral savings (discipline the runtime enforces).

Structural — always on

  • Prompt caching: reuse the unchanged prefix; cache-read ≈ 10% of input cost (R1)
  • Tiered memory: <200-line index + on-demand topic files (R3)
  • Repo map instead of whole files (R4)
  • Diff edit formats: return only changed lines (R4)
  • Deferred MCP tool defs: only names until used (R1)

Behavioral — runtime-enforced

  • Lock model + effort at session start; warn on mid-task model switch (cache invalidation) (R1)
  • Auto-compaction + context editing: evict stale tool results near the limit (R3)
  • Top-k retrieval, dedup, keep deltas not full history (R3)
  • Model tiering: cheap workers (Haiku-class), strong synthesizer only where it counts (R1)
  • Sub-agent isolation: workers compress results back, keeping orchestrator lean (R5)

The cost honesty rule

Multi-mind modes increase total tokens (≈4–15×), even as they reduce orchestrator context. OB-1 shows a live per-mode token budget and post-run cost, and the eval harness must show a mode beating Solo at equal tokens before OB-1 ever auto-suggests it (R5).

08Workspace intelligence

Repository map

Parse with tree-sitter → ASTs of every definition; rank with a PageRank-style graph over the file-dependency graph; inject only the most-referenced symbols that fit the budget. SQLite-cached, incremental, 100+ languages (R4 — the Aider pattern).

Project memory at init

On first run OB-1 generates AGENTS.md: build/test commands, conventions, and architecture notes it discovers — the converging cross-tool standard (R6). Detailed notes spill into on-demand topic files referenced from the index.

09Tools, skills & safety

MCP-native tools

First-class MCP client (stdio/http/sse) — the de-facto standard across Cline, Goose, Gemini CLI, Crush (R6). Tool defs deferred until used. Prefer plain CLI tools (gh, git) where they're more context-efficient (R1).

On-demand skills

A Skills system loaded only when invoked — specialized playbooks live outside base context to keep it small (R1). Skills compose with every mode.

Gated autonomy

Plan/Act modes, per-action approval, auto-approve allowlists with destructive-command gating, and OS sandbox (Seatbelt / bwrap+seccomp), network-off by default (R6 — Cline + Codex CLI).

10Implementation roadmap

Ship the workhorse first; add modes only once the foundation and the eval harness exist (so each mode can be proven, not assumed).

PHASE 0

Solo core. CLI/TUI, gated loop, diff edit engine, approval, provider gateway.

PHASE 1

Context engine. Repo map, tiered memory, JIT+top-k, compaction, token meter.

PHASE 2

Tools. MCP client, Skills, sandbox + permissions, AGENTS.md.

PHASE 3

Multi-mind runtime. Parallel spawner, isolated worker contexts, result compression.

PHASE 4

Fusion + auto-scoring + synthesizer.

PHASE 5

Council + lens-specialized critics + arbiter.

PHASE 6

Personas + blackboard + dynamic former.

PHASE 7

Eval harness. Compute-matched SWE-bench-style proof each mode beats Solo.

The one-sentence thesis

OB-1 wins not by always throwing more models at a problem, but by being a ruthlessly token-frugal single agent by default that can summon Fusion, Council, or Personas when — and only when — the task is worth the spend, with an eval harness that keeps everyone honest.
